# Правила контрибьюта — mh-ts-manager

> Данный документ описывает процесс внесения изменений в репозиторий, стандарты оформления и требования к Pull Request.

---

## 📋 1. Общий процесс

1. **Форк** (если внешний контрибьютор) или **клонирование** репозитория
2. **Создание ветки** от `main` по правилам именования (см. раздел 2)
3. **Внесение изменений** с соблюдением [RULES.md](docs/development/RULES.md)
4. **Тестирование** — все тесты должны проходить
5. **Коммиты** по правилам оформления (см. раздел 3)
6. **Pull Request** в `main` с подробным описанием (см. раздел 4)
7. **Ревью** — минимум один approve от мейнтейнера
8. **Мерж** после прохождения всех проверок

---

## 🌿 2. Ветки

### 2.1. Формат именования

| Тип | Формат | Пример |
|-----|--------|--------|
| Новая функциональность | `feature/описание` | `feature/column-drag-drop` |
| Исправление ошибки | `fix/описание` | `fix/session-state-aggregation` |
| Рефакторинг | `refactor/описание` | `refactor/wts-service-di` |
| Документация | `docs/описание` | `docs/update-rules-md` |
| Тесты | `test/описание` | `test/window-filter-tests` |
| Оптимизация | `perf/описание` | `perf/session-cache-lock` |
| Технические задачи | `chore/описание` | `chore/update-gitignore` |

### 2.2. Правила

- ❌ **Запрещена** прямая работа с `main`
- ❌ **Запрещён** force-push в `main`
- ✅ Ветки создаются **от актуального `main`**
- ✅ Ветки **удаляются** после мержа

## 📝 3. Коммиты

### 3.1. Формат

```
тип: Краткое описание (до 72 символов)

Подробное описание (опционально, через пустую строку).
Максимальная ширина — 100 символов на строку.
```

### 3.2. Типы коммитов

| Тип | Описание | Пример |
|-----|----------|--------|
| `feat` | Новая функциональность | `feat: добавление drag-and-drop для столбцов` |
| `fix` | Исправление ошибки | `fix: корректное определение заблокированной сессии` |
| `refactor` | Рефакторинг без изменения поведения | `refactor: выделение SafeHandle для WTS API` |
| `perf` | Оптимизация производительности | `perf: кэширование иконок приложений` |
| `docs` | Изменения в документации | `docs: обновление технической спецификации` |
| `test` | Добавление/изменение тестов | `test: юнит-тесты для WindowEnumeratorService` |
| `chore` | Технические изменения | `chore: обновление .gitignore для WinUI` |
| `ci` | Изменения в CI/CD | `ci: добавление GitHub Actions workflow` |
| `style` | Форматирование (без логики) | `style: исправление отступов в XAML` |

### 3.3. Язык

- **Язык коммитов:** русский
- **Начало:** строчная буква, без точки в конце
- **Залог:** совершенный вид («добавление», «исправление», «обновление»)

### 3.4. Примеры

```
✅ feat: добавление локализации для de-DE
✅ fix: утечка хендлов при EnumWindows
✅ refactor: замена lock на ReaderWriterLockSlim
✅ docs: обновление правил контрибьюта
✅ test: покрытие SettingsService тестами
✅ perf: переиспользование буферов через ArrayPool

❌ "update" — слишком общее
❌ "fix bugs" — неясно, что исправлено
❌ "wip" — промежуточный коммит (squash перед PR)
❌ "Update README.md" — английский язык
```

### 3.5. Squash

Перед отправкой PR рекомендуется squash промежуточных коммитов:

```powershell
# Интерактивный rebase для squash
git rebase -i HEAD~5
```

## 📬 4. Pull Request

### 4.1. Заголовок

Формат: `[Тип]: Краткое описание на русском`

```
✅ [feat]: добавление drag-and-drop для столбцов таблицы
✅ [fix]: корректное определение заблокированной сессии
```

### 4.2. Описание

**Обязательная структура:**

```markdown
## 📋 Проблема

Описание проблемы, симптомы, шаги для воспроизведения.

## ✅ Решение

Что было сделано, какие файлы затронуты, подход к решению.

## 📊 Результат

Как проверить, что всё работает. Скриншоты, примеры вывода.

## ✅ Тестирование

- [ ] Сборка без ошибок (`dotnet build -c Release`)
- [ ] Все unit-тесты пройдены (`dotnet test`)
- [ ] Ручное тестирование на Windows 10
- [ ] Ручное тестирование на Windows 11
- [ ] Тестирование локализации (ru-RU, en-US)
```

### 4.3. Чек-лист перед отправкой

- [ ] Код следует [RULES.md](docs/development/RULES.md)
- [ ] Все тесты проходят
- [ ] Нет предупреждений анализаторов
- [ ] Коммиты squash-нуты, имена соответствуют формату
- [ ] Описание PR подробное, на русском языке
- [ ] Раздел `## ✅ Тестирование` заполнен
- [ ] Затронутые файлы перечислены с пояснениями

### 4.4. Размер PR

- **Предпочтительно:** небольшие PR (1-3 файла, одна задача)
- **Максимум:** 15 файлов, 500 строк изменений
- **Крупные изменения:** разбивать на несколько PR

## 🐛 5. Баг-репорты

При создании баг-репорта (Issue) указать:

1. **Версия ПО:** `mh-ts-manager v1.3.0`
2. **Версия ОС:** Windows 10 22H2 / Windows 11 23H2
3. **Шаги воспроизведения:** пошаговое описание
4. **Ожидаемый результат:** что должно произойти
5. **Фактический результат:** что произошло
6. **Логи:** фрагмент из `%APPDATA%\mh-ts-manager\logs\` (без PII)
7. **Скриншоты:** если проблема визуальная

---

## 💡 6. Запрос новых функций

Перед запросом новой функциональности:

1. Проверить существующие Issues и Discussions
2. Описать **use-case**: зачем это нужно, какую проблему решает
3. Предложить **подход к реализации** (если возможно)
4. Указать **приоритет**: критично / желательно / nice-to-have

---

## 🔒 7. Безопасность

При обнаружении уязвимости:

- ❌ **Не создавать** публичный Issue с деталями
- ✅ Написать мейнтейнеру напрямую (GitHub Private Vulnerability Reporting)
- ✅ Предложить шаги для исправления (если возможно)
